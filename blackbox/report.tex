\section{Objective}
This part of the assignment performs black-box testing on the QuickCart REST API using only the given API documentation. The internal implementation was not available. The testing focused on request validation, response structure, status codes, returned data, and user-visible business rules.

\section{Approach}
Two kinds of testing were used.

First, manual exploratory testing was used to understand endpoint behavior from the documentation, inspect seeded data through the admin endpoints, and identify likely edge cases.

Second, automated tests were written using only \texttt{pytest} and \texttt{requests}. These tests check:
\begin{itemize}
    \item correct HTTP status codes,
    \item JSON response structure,
    \item correctness of values returned by the API,
    \item valid requests,
    \item invalid inputs,
    \item missing fields,
    \item wrong data types,
    \item boundary values.
\end{itemize}

The automated suite targets the API at \texttt{http://127.0.0.1:8080/api/v1}. Admin requests use the \texttt{X-Roll-Number} header, and user-scoped requests use both \texttt{X-Roll-Number} and \texttt{X-User-ID}. The final submission uses the student's roll number, \texttt{2024111034}, in the test configuration and documented request examples.

\section{Automated Execution Summary}
The final automated run for the submitted black-box suite produced the following result:
\begin{itemize}
    \item Total automated test cases: 44
    \item Passed: 36
    \item Failed: 8
\end{itemize}

The failing tests correspond to real API bugs, not test setup issues.

\section{Why Both Manual and Automated Testing Were Used}
In black-box testing, test cases are first designed from the API specification without using internal code. Manual testing helps understand how the API behaves in practice, discover seeded data values, and identify useful edge cases. Automated testing then makes those checks repeatable and objective. For this assignment, both were useful:
\begin{itemize}
    \item manual testing helped understand the endpoints and data,
    \item automated pytest cases produced repeatable evidence,
    \item the final report is based on the automated results and the observed API behavior.
\end{itemize}

\section{Bug Report}
The following bugs were reproduced by the automated test suite.

\subsection{Bug 1: Valid 6-digit pincode is rejected}
\textbf{Endpoint tested:} \texttt{POST /api/v1/addresses}

\textbf{Request payload:}
\begin{itemize}
    \item Method: \texttt{POST}
    \item URL: \texttt{http://127.0.0.1:8080/api/v1/addresses}
    \item Headers: \texttt{X-Roll-Number: 2024111034}, \texttt{X-User-ID: 1}
    \item Body: \texttt{\{"label":"OTHER","street":"12345 Test Street <id>","city":"Delhi","pincode":"110011","is\_default":false\}}
\end{itemize}
\textbf{Expected result:} Address should be created successfully with \texttt{200}.\\
\textbf{Actual result:} API returned \texttt{400} with \texttt{\{"error":"Invalid pincode"\}}.

\subsection{Bug 2: Cart accepts quantity 0}
\textbf{Endpoint tested:} \texttt{POST /api/v1/cart/add}

\textbf{Request payload:}
\begin{itemize}
    \item Method: \texttt{POST}
    \item URL: \texttt{http://127.0.0.1:8080/api/v1/cart/add}
    \item Headers: \texttt{X-Roll-Number: 2024111034}, \texttt{X-User-ID: 1}
    \item Body: \texttt{\{"product\_id":1,"quantity":0\}}
\end{itemize}
\textbf{Expected result:} API should reject quantity 0 with \texttt{400}.\\
\textbf{Actual result:} API returned \texttt{200} and added the item.

\subsection{Bug 3: Cart item subtotal is calculated incorrectly}
\textbf{Endpoint tested:} \texttt{GET /api/v1/cart} after adding the same product twice

\textbf{Request payload:}
\begin{itemize}
    \item Method: \texttt{GET}
    \item URL: \texttt{http://127.0.0.1:8080/api/v1/cart}
    \item Headers: \texttt{X-Roll-Number: 2024111034}, \texttt{X-User-ID: 1}
    \item Body: none
\end{itemize}
\textbf{Expected result:} For quantity 2 and unit price 120, subtotal should be 240.\\
\textbf{Actual result:} API returned subtotal \texttt{-16}.

\subsection{Bug 4: Cart total is calculated incorrectly}
\textbf{Endpoint tested:} \texttt{GET /api/v1/cart} after adding two different products

\textbf{Request payload:}
\begin{itemize}
    \item Method: \texttt{GET}
    \item URL: \texttt{http://127.0.0.1:8080/api/v1/cart}
    \item Headers: \texttt{X-Roll-Number: 2024111034}, \texttt{X-User-ID: 1}
    \item Body: none
\end{itemize}
\textbf{Expected result:} Cart total should equal the sum of all item subtotals.\\
\textbf{Actual result:} API returned total \texttt{120} while item subtotals summed to \texttt{-6}.

\subsection{Bug 5: Wallet payment deducts more than requested}
\textbf{Endpoint tested:} \texttt{POST /api/v1/wallet/pay}

\textbf{Request payload:}
\begin{itemize}
    \item Method: \texttt{POST}
    \item URL: \texttt{http://127.0.0.1:8080/api/v1/wallet/pay}
    \item Headers: \texttt{X-Roll-Number: 2024111034}, \texttt{X-User-ID: 1}
    \item Body: \texttt{\{"amount":1\}}
\end{itemize}
\textbf{Expected result:} Wallet balance should decrease by exactly 1.\\
\textbf{Actual result:} The observed balance dropped by more than 1, showing incorrect deduction logic.

\subsection{Bug 6: Checkout allows an empty cart}
\textbf{Endpoint tested:} \texttt{POST /api/v1/checkout}

\textbf{Request payload:}
\begin{itemize}
    \item Method: \texttt{POST}
    \item URL: \texttt{http://127.0.0.1:8080/api/v1/checkout}
    \item Headers: \texttt{X-Roll-Number: 2024111034}, \texttt{X-User-ID: 1}
    \item Body: \texttt{\{"payment\_method":"COD"\}}
\end{itemize}
\textbf{Expected result:} API should reject empty-cart checkout with \texttt{400}.\\
\textbf{Actual result:} API returned \texttt{200} and created an order.

\subsection{Bug 7: Review average is not returned as a proper decimal}
\textbf{Endpoint tested:} \texttt{GET /api/v1/products/\{id\}/reviews} after adding a new review

\textbf{Request payload:}
\begin{itemize}
    \item Method: \texttt{GET}
    \item URL: \texttt{http://127.0.0.1:8080/api/v1/products/1/reviews} (product id varies by test data)
    \item Headers: \texttt{X-Roll-Number: 2024111034}, \texttt{X-User-ID: 1}
    \item Body: none
\end{itemize}
\textbf{Expected result:} Average rating should be the true decimal average.\\
\textbf{Actual result:} API returned a truncated integer value such as \texttt{3} instead of the correct decimal average.

\subsection{Bug 8: Support ticket workflow allows backward status transition}
\textbf{Endpoint tested:} \texttt{PUT /api/v1/support/tickets/\{ticket\_id\}}

\textbf{Request payload:}
\begin{itemize}
    \item Method: \texttt{PUT}
    \item URL: \texttt{http://127.0.0.1:8080/api/v1/support/tickets/<ticket\_id>}
    \item Headers: \texttt{X-Roll-Number: 2024111034}, \texttt{X-User-ID: 1}
    \item Body: \texttt{\{"status":"OPEN"\}} after the ticket is already in \texttt{IN\_PROGRESS}
\end{itemize}
\textbf{Expected result:} API should reject the backward transition with \texttt{400}.\\
\textbf{Actual result:} API returned \texttt{200} and allowed the status change.

\section{Conclusion}
The QuickCart black-box suite covers valid inputs, invalid inputs, missing fields, wrong data types, and boundary values using only the published API behavior. The final automated run found 8 confirmed API bugs while 36 other tests passed successfully. This shows that the API has many correct validation paths, but still contains important defects in address validation, cart calculation, wallet deduction, checkout validation, review averaging, and support-ticket workflow.
